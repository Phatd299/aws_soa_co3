## DOMAIN 1: MONITORING, LOGGING, AND REMEDIATION

CloudWatch Metrics. EC2 sends metrics at 5-minute intervals by default, 1-minute with detailed monitoring enabled. Memory utilization, disk space, and disk I/O are NOT available by default for EC2, you must install the CloudWatch Agent to get those. This is one of the most tested facts in the exam, any scenario involving memory or disk alarms on EC2 requires the agent. Custom metrics can be pushed via put-metric-data. High-resolution metrics support down to 1 second. Metric Math lets you combine metrics with expressions without creating new custom metrics. Anomaly detection creates a band based on historical data, useful when static thresholds don't fit a variable workload.

CloudWatch Logs. Logs Insights is a query tool, not a transformation tool, and it does not support modifying or exporting in real time, only ad hoc querying. Subscription Filters stream log data in near real time to Kinesis, Firehose, or Lambda for further processing, this is the answer when a question asks for real time processing of incoming logs. Contributor Insights analyzes log data to surface top talkers, like which IP is generating the most errors, without you writing custom queries. Retention is not infinite by default, default is "never expire" but you should set explicit retention periods to manage cost. Logs can be exported to S3 for long term archival, but that export is batch, not real time, again pointing to Subscription Filters when real time matters.

CloudWatch Alarms. M-of-N datapoint evaluation lets you say "trigger if 3 out of 5 most recent datapoints breach the threshold" which smooths out noisy metrics. Missing data treatment has four options: treat as missing (default, does not affect alarm state), treat as breaching, treat as not breaching, or ignore (maintains current state). This matters for instances that stop reporting, like a crashed instance, where you'd want missing data treated as breaching to catch the failure. Composite Alarms combine multiple alarms with AND/OR logic but cannot reference other composite alarms, only standard metric alarms, and they help reduce alarm noise/flapping.

CloudTrail. Records API calls (management events and optionally data events like S3 object level activity). There is a roughly 15 minute delivery lag to S3 for the trail's log files, so if a question describes wanting near instant visibility into an API call, CloudTrail to S3 is not the fast path, CloudTrail Lake or EventBridge integration is faster. Log file integrity validation provides tamper resistant proof using digest files and hashing. CloudTrail is regional by default unless you configure a multi region trail, and a single trail can also be organization wide in AWS Organizations.

X-Ray. Used for distributed tracing across microservices. A trace is the end to end journey of a request, segments are the work done by one service, subsegments break that down further like a downstream call. Sampling is the first lever to pull when troubleshooting cost or volume issues with X-Ray, since 100% tracing is rarely necessary and the default sampling rule is 1 request/second plus 5% of additional requests.

EventBridge. The key differentiator versus SNS and SQS is content based routing/filtering, EventBridge can route events based on the content of the event itself using rules, without the publisher needing to know consumers, and without needing a fan out from a single topic. SNS is pub/sub with fan out to multiple subscribers (email, SMS, SQS, Lambda) but does not do content based filtering on a granular level the way EventBridge rules do. SQS is point to point queuing for decoupling producers and consumers, with messages pulled rather than pushed, useful when you want guaranteed processing and buffering. If you see "route based on event content/pattern" think EventBridge. If you see "decouple and buffer for asynchronous processing with guaranteed delivery to one consumer" think SQS. If you see "broadcast to multiple subscribers" think SNS.

## DOMAIN 2: RELIABILITY AND BUSINESS CONTINUITY

RDS. Multi-AZ provides synchronous replication to a standby in a different AZ for high availability and automatic failover, it is not a read scaling solution. Read Replicas are asynchronous and used for read scaling, and can be promoted to standalone but that is a manual, deliberate action, not automatic failover. Multi-AZ and Read Replicas are not mutually exclusive, many production designs use both, Multi-AZ for HA and Read Replicas for offloading read traffic. Point in Time Recovery, PITR, relies on automated backups and transaction logs and lets you restore to any point within the backup retention window (up to 35 days), this restores to a NEW instance, it does not restore in place. Automated backups happen daily during the backup window and can cause elevated I/O latency on Single-AZ deployments during the backup snapshot moment, this is a frequently tested gotcha, the fix is Multi-AZ since the backup is taken from the standby.

Backup strategies generally. AWS Backup centralizes backup policies across many services (EBS, RDS, DynamoDB, EFS, etc.) via backup plans and lifecycle rules, that is the answer when a question wants centralized, automated, cross-service backup management. Cross region backup copies and cross region replication (like S3 CRR) are the answer for disaster recovery scenarios needing data in a second region.

DynamoDB. On-Demand Backup is full table backups, no performance impact, manual or scheduled via AWS Backup. PITR on DynamoDB is continuous backups enabling restore to any second within the last 35 days, separate feature from on-demand backup, you can have both enabled. Global Tables provide multi region, active active replication for DynamoDB, this is the DR/HA answer for DynamoDB specifically, not a backup feature but a replication feature.

Route 53 and failover. Routing policies: simple, weighted, latency based, failover, geolocation, geoproximity, multivalue answer. Failover routing requires health checks to detect when the primary is unhealthy and shift to secondary, health checks alone do nothing without a routing policy wired to them. Latency based routing routes to the region with lowest latency for the user, it is not failover, but is often layered with health checks to remove unhealthy endpoints from being routed to.

Auto Scaling. Scaling policies: target tracking (maintain a metric at a target value, generally recommended as least operational overhead), step scaling (scale by defined increments based on alarm breach size), simple scaling (one adjustment, then cooldown). Lifecycle hooks let you pause an instance in Pending or Terminating state to run custom actions, like draining connections or pulling logs before termination. Health checks for ASG can be EC2 status checks or ELB health checks, ELB health checks are needed if you want app level health (like a 200 OK on a path) rather than just instance level reachability.

ELB. Application Load Balancer operates at layer 7, supports path and host based routing, useful for microservices. Network Load Balancer operates at layer 4, used for extreme performance/low latency or static IP requirements, and is the correct choice when a question mentions needing a static IP or Elastic IP on the load balancer or handling millions of requests per second. Cross zone load balancing distributes evenly across all registered targets in all AZs, important when AZs have an uneven number of targets.

Disaster recovery strategies, the four tiers, ordered by cost and RTO/RPO tradeoff: Backup and Restore (cheapest, slowest RTO), Pilot Light (minimal core infra always running, scaled up on failover), Warm Standby (scaled down but fully functional version always running, faster failover than pilot light), Multi-Site Active/Active (most expensive, near zero RTO/RPO, full capacity in multiple regions simultaneously). Exam scenarios describe the business requirement (cost sensitivity vs RTO/RPO targets) and you match to the tier.

## DOMAIN 3: DEPLOYMENT, PROVISIONING, AND AUTOMATION

CloudFormation. Declarative infrastructure as code. Stack drift detection identifies when actual resource configuration has diverged from what the template defines, this is the answer when a question asks how to detect manual/out of band changes to CloudFormation managed resources. Change sets let you preview what will happen before executing an update, the safe way to validate impact before applying. StackSets deploy the same stack across multiple accounts and regions, the answer for multi account/multi region consistent deployment. Nested stacks let you break large templates into reusable components. DeletionPolicy and UpdateReplacePolicy attributes control whether a resource (like an S3 bucket or RDS instance) is retained, snapshotted, or deleted when a stack resource is removed or replaced, frequently tested with the Retain value to avoid accidental data loss.

Elastic Beanstalk. PaaS layer for deploying applications without managing the underlying infrastructure directly, still gives you access to underlying resources if needed. Deployment policies: All at once (fastest, downtime), Rolling (batches, no full downtime but reduced capacity during deploy), Rolling with additional batch (maintains full capacity by launching new batch before taking old ones down), Immutable (full new environment of new instances, safest rollback, most expensive/slowest), Blue/Green isn't a built in EB policy name but is achieved via environment swap (CNAME swap between two environments), this is the best option for zero downtime with instant rollback capability.

Systems Manager, SSM. Run Command executes commands across fleets of instances without SSH. Patch Manager automates OS/application patching on a schedule. State Manager maintains a defined configuration state over time, correcting drift automatically. Parameter Store holds configuration data and secrets (standard tier free, advanced tier for larger values/higher throughput/parameter policies). Secrets Manager is the choice over Parameter Store when you need automatic rotation of credentials, particularly for RDS/database credentials, this is the most common differentiator tested, Parameter Store does not natively auto rotate, Secrets Manager does.

OpsWorks is legacy Chef/Puppet based config management, low exam weight, know it exists as the older alternative to SSM but expect minimal direct testing.

CodeDeploy. Deploys application code to EC2, on-premises, Lambda, or ECS. Deployment configurations include AllAtOnce, HalfAtATime, OneAtATime, and custom. AppSpec file defines deployment steps and hooks. Supports blue/green natively for EC2/ASG and Lambda, this is relevant when a question wants automated blue/green without manually swapping environments like in Beanstalk.

Auto Scaling launch templates vs launch configurations: launch configurations are legacy and immutable (cannot be edited, must create new), launch templates are the modern recommended approach, support versioning, and allow mixing instance types and purchase options (Spot + On-Demand) within one ASG, a frequently tested point when cost optimization scenarios appear.

## DOMAIN 4: SECURITY AND COMPLIANCE

IAM. Policies are evaluated with explicit deny always winning over any allow, no policy present means implicit deny. Permission boundaries set the maximum permissions an identity can have regardless of attached policies, used to constrain what delegated administrators can grant. Service Control Policies, SCPs, apply at the AWS Organizations level and restrict accounts even if an account's own IAM policies would allow more, SCPs never grant permissions, only restrict.

GuardDuty vs Inspector vs Config, a heavily tested trio. GuardDuty is continuous threat detection using logs (VPC Flow Logs, DNS logs, CloudTrail) to detect things like unusual API calls or compromised credentials, it is detection only, it does not fix anything. Inspector is vulnerability assessment for EC2 and container images and Lambda, scanning for software vulnerabilities and unintended network exposure, also detection only. AWS Config records configuration state and compliance against rules over time and CAN trigger automatic remediation actions via Config Rules with SSM Automation documents, Config is the one of these three that can actually remediate, the others only alert/detect.

Security Groups vs NACLs. Security groups are stateful (return traffic automatically allowed), operate at the instance/ENI level, support only allow rules. NACLs are stateless (return traffic must be explicitly allowed by a rule), operate at the subnet level, support both allow and deny rules, and are evaluated in rule number order. The "deny specific IP" scenario always points to NACLs since security groups cannot deny.

KMS. Customer managed keys give you control over key policy and rotation settings, AWS managed keys are used by services by default and rotate automatically with rotation managed by AWS, AWS owned keys are not visible/manageable by you at all. Envelope encryption is the underlying mechanism, a data key encrypts the data and the data key itself is encrypted by the KMS key (CMK). Key policies are the primary access control for KMS keys, IAM policies alone are not sufficient unless the key policy also allows IAM policy based control via the default key policy statement.

AWS Organizations and multi account security. SCPs apply org wide or per OU. Consolidated billing and centralized logging (CloudTrail org trail, centralized GuardDuty/Security Hub) are common "least operational overhead across many accounts" answers.

WAF and Shield. WAF protects against layer 7 attacks like SQL injection and XSS using rules/rule groups, attached to CloudFront, ALB, or API Gateway. Shield Standard is automatic and free for all customers, covering common layer 3/4 DDoS. Shield Advanced is paid, covers larger/more sophisticated attacks, gives cost protection during DDoS events, and includes access to the DDoS Response Team.

Secrets and credentials. Secrets Manager for anything needing rotation, especially database credentials, Parameter Store for general config and secrets without that native rotation need, this distinction comes up constantly in cost optimization vs security scenarios.

## DOMAIN 5: NETWORKING AND CONTENT DELIVERY

VPC fundamentals. Each subnet lives in one AZ. Route tables control traffic at the subnet level, a subnet with a route to an Internet Gateway is public, without one it's private. NAT Gateways allow outbound internet access for private subnets, are AZ scoped (deploy one per AZ for high availability, a single NAT Gateway is a single point of failure across AZs), and are managed/scaled by AWS unlike NAT instances which you'd manage yourself, NAT instances are legacy/rarely the right exam answer unless cost is explicitly the overriding constraint and the question wants you to know the tradeoff.

VPC Peering vs Transit Gateway. Peering is point to point, does not support transitive routing (A peered to B peered to C does not let A talk to C), this lack of transitivity is the single most tested VPC peering fact. Transit Gateway is a hub and spoke model supporting many VPCs and on-premises connections with transitive routing built in, the answer whenever a question describes connecting many VPCs together or needing transitive connectivity.

VPC Endpoints. Gateway endpoints support only S3 and DynamoDB, free, implemented via route table entries. Interface endpoints (powered by PrivateLink) support most other AWS services, have an hourly cost plus data processing cost, appear as an ENI with a private IP in your subnet. Both let you reach AWS services without traversing the public internet/NAT Gateway, the answer whenever a scenario wants private connectivity to AWS services without internet exposure.

VPN and Direct Connect. Site to Site VPN is encrypted, runs over the public internet, faster to set up, variable latency. Direct Connect is a dedicated physical network connection, consistent low latency and higher bandwidth, but takes weeks/months to provision, the answer when a question describes need for consistent, high bandwidth, low latency connectivity to on-premises and time to provision is not the constraint. Direct Connect plus VPN as a backup is a common HA pattern, tested as the answer for "want DX performance but need failover."

CloudFront. Origin can be S3 (use Origin Access Control, OAC, the modern replacement for the older OAI, to restrict S3 access to only CloudFront) or any HTTP origin like an ALB. Cache behaviors let you route different paths to different origins and control caching/TTL per path pattern. Signed URLs/cookies restrict content access, signed URLs for individual files, signed cookies when you want to grant access to multiple restricted files at once (like a whole subscription protected site section).

Global Accelerator vs CloudFront. CloudFront is a content delivery/caching service, optimized for cacheable HTTP/S content. Global Accelerator improves performance for non cacheable, dynamic, or non HTTP traffic (like gaming, IoT, VoIP) by routing over the AWS global network at the TCP/UDP layer, the differentiator the exam wants is "is the traffic cacheable web content" point to CloudFront, "is the traffic dynamic/non HTTP needing static anycast IPs and network layer optimization" point to Global Accelerator.

Hybrid DNS and Route 53 Resolver. Route 53 Resolver endpoints (inbound and outbound) allow DNS resolution between on-premises and VPC, inbound endpoint lets on-prem resolve VPC private hosted zone records, outbound endpoint lets VPC resources resolve on-premises DNS via conditional forwarding rules.

VPC Flow Logs. Capture metadata about traffic (source/dest IP, port, protocol, accept/reject) not the actual packet content/payload, used for troubleshooting connectivity issues (like an unexpected REJECT) and security analysis, feeds into GuardDuty as one of its primary detection data sources.
